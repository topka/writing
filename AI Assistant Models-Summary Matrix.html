AI Personal Assistant Models — Summary Evaluation

Each model evaluated from three perspectives. Working draft, March 2026.

Model Employee's Perspective Organization's Concerns Practical Viability
Single-Assistant Models
M1 Fully unifiedOne assistant, one memory, one context, no boundaries. Severe
Worst case — all personal data exposed, no control over whose interests the assistant serves, no recourse on departure.		 Adequate
Attractive — full control, maximum productivity potential, simple to manage. But holding personal data creates new regulatory exposure.		 Strong
Trivially buildable. This is what the market is actually shipping today.
M2 Soft separationOne assistant, one system, policy-based rules governing cross-domain data flow. Problematic
Better than M1 but depends entirely on who writes the policies and whether they hold. Data still co-exists in one system, so protection is only as good as enforcement.		 Adequate
Workable — retains control, can set policies, supports audit. But policy complexity grows fast and personal data still complicates compliance.		 Adequate
Moderate complexity. DLP and classification tools exist; applying them to AI context is new but doable.
M3 Hard partitionOne assistant interface, two separate memory stores, user-controlled bridge. Mixed
Real architectural protection. But one interface still means one optimization function — the agent's loyalty is ambiguous even if the data is cleanly separated.		 Mixed
Partial control — governs the work partition but can't see the personal side, and the user-controlled bridge is a potential leak vector.		 Mixed
Significant engineering that nobody has built yet. User must actively manage which context they're in, adding cognitive overhead.
Single-Assistant Models — Flipped Ownership
M4 User-ownedIndividual owns the assistant; employer grants access to work systems. Strong
Strongest model for the individual — full ownership, full control, knowledge stays on departure.		 Severe
Near-unacceptable — an uncontrolled AI agent accessing corporate data with no audit trail, no governance, no ability to restrict or configure.		 Problematic
Few organizations would allow this today. BYOD-for-AI is not an accepted practice.
M5 Company-ownedEmployer owns the assistant; employee optionally connects personal data and tools. Problematic
Familiar pattern (like a company laptop) but structurally problematic — data enters a system the employee doesn't control, and the agent's optimization serves the employer regardless of personal extensions.		 Strong
Strongest model for the company — full ownership, full audit, full control over configuration and optimization.		 Strong
Closest to current market reality. Essentially what Copilot and Glean are evolving toward.
Two-Assistant Models
M6 Two assistants, user-managedSeparate assistants in a shared user environment; user bridges with low friction. Strong
Clean separation with easy bridging — both assistants live in a shared environment, so passing information is low-friction while still intentional and user-controlled.		 Adequate
Clean and controllable — work assistant is fully org-managed, and the shared environment means some governance of cross-domain interactions is possible.		 Adequate
Two separate products on the user's own device, coordinating via standard protocols. No shared platform needed — just protocol adoption and smooth integration UX.
M7 Two assistants, AI-mediatedSeparate assistants plus a third AI coordination layer that brokers cross-domain interactions. Adequate
Best balance of separation and intelligence — real domain boundaries with an AI broker that can surface conflicts and pass selective information without manual effort.		 Adequate
Reasonable — work assistant stays org-controlled, coordination layer is a scoped interface. But the coordinator is a new system to audit and govern.		 Problematic
Protocol building blocks exist in production (A2A + DIDComm V2 + VC 2.0 with ZKPs/SD-JWTs). But no consumer product yet — existing vendors focus on enterprise, not personal assistant coordination.
M8 Two assistants, complete separationSeparate environments, separate devices, no shared surface. Like two phones. Mixed
Maximum protection but maximum inconvenience — separate devices with no shared surface, like carrying a work phone and a personal phone that never touch. Protection is strong, but the UX cost undermines the value of having a personal assistant at all.		 Adequate
Perfect isolation and control. But the work assistant has zero personal context, limiting holistic productivity — the org pays for a "personal assistant" that can only assist with work tasks.		 Mixed
Already the default reality — just using two unrelated AI products. But the UX of all-manual bridging defeats much of the personal assistant purpose.
